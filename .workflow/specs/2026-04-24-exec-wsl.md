# Exec: WSL-execution + selective sync roadmap

**Date:** 2026-04-24
**Scope:** Commands tab (`src/tabs/exec.js` + `src-tauri/src/commands/exec.rs`)

## Phase 1 — WSL execution (in progress)

### Проблема

Сейчас `run_command` хардкодит `tokio::process::Command::new("sh").arg("-c")`.
На Windows без git-bash `sh.exe` в PATH это не работает. Плюс юзер
хочет запускать команды внутри WSL чтобы:
- использовать `~/.ssh/config` и ключи, которые лежат в WSL-home
- использовать WSL-пути (`/home/user/...`)
- иметь Linux-окружение (bash, rsync, scp, jq и т.п.)

### Решение

Добавить per-command selector **Shell**: `host | wsl`.
При `wsl` — опциональный выбор дистро из автоподтянутого списка.

### Модель БД (миграция)

```sql
ALTER TABLE exec_commands ADD COLUMN shell TEXT NOT NULL DEFAULT 'host';
ALTER TABLE exec_commands ADD COLUMN wsl_distro TEXT;    -- NULL = default
```

ALTER-ы идемпотентны через `.ok()` (существующий паттерн в `db/mod.rs`).

### Раскладка запуска

```
shell='host':
  cfg(windows) → Command::new("cmd").args(["/c", cmd])
  else        → Command::new("sh").arg("-c").arg(cmd)

shell='wsl' (только Windows; Mac/Linux → ошибка "WSL only on Windows"):
  args = ["wsl.exe"]
  if let Some(d) = wsl_distro { args += ["-d", &d]; }
  args += ["--", "bash", "-lc", escaped_cmd]
  //  `bash -lc` (login shell) — подхватывает ~/.bashrc, PATH, ssh-agent
  //  escaped_cmd: `'` → `'\''` для безопасной вставки внутрь bash -lc '…'
```

### Backend

- **Новая команда** `list_wsl_distros() -> Vec<String>`
  - На Windows: `wsl.exe -l -q`, парсит UTF-16LE stdout, filter empty
  - На Mac/Linux: `Ok(vec![])`
- **`run_command(command, shell, wsl_distro)`** — новая сигнатура,
  старые вызовы получают `shell='host', wsl_distro=None` через
  frontend-адаптер (либо новая Tauri-команда + пометить старую
  `run_command` deprecated, либо расширить сигнатуру с optional
  параметрами)
- **`CREATE_NO_WINDOW` flag** на Windows (как в `repo_search.rs:22`) —
  иначе каждый запуск светит cmd-окно

### Frontend

- В карточке команды (создание/редактирование):
  - Новый dropdown **Shell** с опциями `Host` / `WSL`
  - Если `WSL` → рядом появляется dropdown **Distro**
    с опцией `(default)` первой + список из `list_wsl_distros()`
- На Mac/Linux — dropdown показывает только `Host`, `WSL` disabled
  с hint "Windows only"

### Тесты

- Unit: escape `'` в пользовательской команде → `'\''`
- Unit: `wsl_args(None)` vs `wsl_args(Some("Ubuntu-22.04"))`
- Smoke: запуск `echo hello` через host на всех платформах
- Smoke (Windows): `wsl -- echo hello` возвращает "hello"

### UX-решения (утверждены 2026-04-24)

| Q | Решение |
|---|---------|
| Q1 cwd | Не добавляем поле — юзер пишет `cd /path && cmd` inline |
| Q2 granularity | Per-command (не per-category) |
| Q3 distro list | Auto-detect через `wsl -l -q` |
| Q4 output | Batch (как сейчас), streaming в Phase 3 |
| Q5 no-WSL error | Понятная ошибка; без auto-fallback на host |
| Q6 backward compat | DEFAULT 'host' — все существующие команды работают как раньше |
| Q7 sync | `exec_commands` остаётся local-only (sync в Phase 2) |
| Q8 escaping | bash-escape `'` → `'\''` |

## Phase 2 — Selective command sync + per-machine variables (future)

### Use case

> «Мне нужна команда которая что-то копирует с удалённого хоста, или
> на него. На одном ноутбуке один локальный путь, на другом —
> другой, при этом удалённый путь на самом VPS — одинаковый. Надо
> добавить опцию как-то скопировать команду с одного ноута на
> другой, с попутной модификацией.»

### Решение (эскиз)

1. **Миграция**: добавить опциональный `is_synced BOOLEAN DEFAULT 0` +
   sync-поля (`uuid`, `updated_at`, `sync_status`, `user_id`) в
   `exec_commands`.
2. **Sync schema**: добавить `exec_commands` в `SYNCED_TABLES` +
   `data_columns` (вместе с новыми `shell`, `wsl_distro`, `is_synced`)
3. **Per-machine variables**: расширить команду плейсхолдерами
   `{{VAR_NAME}}` + таблица `exec_variables`:
   ```sql
   CREATE TABLE exec_variables (
     id INTEGER PRIMARY KEY,
     computer_id TEXT NOT NULL,
     var_name TEXT NOT NULL,
     var_value TEXT NOT NULL,
     UNIQUE(computer_id, var_name)
   );
   ```
   НЕ синкается (per-machine).
4. **Resolve at run-time**: перед запуском команды подставляем
   `{{VAR}}` из `exec_variables` для текущей машины.
5. **UI**:
   - Галочка "Sync this command across machines" в редакторе команды
   - Секция "Machine variables" в Settings (экранчик в виде key/value,
     где на каждой машине свой список с своими значениями)
   - Предупреждение "This command uses {{VAR_NAME}} which isn't
     defined on this machine" — с inline-добавлением
6. **Пример**:
   - Команда (sync=true): `scp /home/user/{{PROJECT_DIR}}/backup.sql vps:/srv/backups/`
   - На ноут A: `PROJECT_DIR=work/proj-x`
   - На ноут B: `PROJECT_DIR=home/side-proj`
   - После sync команда появится на обоих ноутах, а на ноуте B
     system предложит добавить `PROJECT_DIR` перед первым запуском.

### Зависимости

- Phase 1 WSL-экзекьюшн должен быть стабилен
- Нужно решить что делать с `exec_categories` — синкать или нет
  (аргумент за: категории тоже могут различаться)

### Открытые вопросы (к обсуждению при старте Phase 2)

- Должны ли `shell` / `wsl_distro` синкаться? Если юзер на Mac создал
  команду с `shell='host'`, а на Windows она подтянется — `host` там
  будет `cmd /c`, не `sh -c`. Потенциальная неочевидность.
  **Возможное решение**: `shell` НЕ синкается (per-machine), default
  computed на receiving machine.
- `wsl_distro` должен быть валидирован на receiving-машине (может не
  быть такой дистро). Отметить команду как broken с hint в UI.
- Как именовать variables — UPPER_SNAKE или CamelCase? Предложить
  UPPER_SNAKE как convention, валидировать в UI.
