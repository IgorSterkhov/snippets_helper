
Как подписывать коммиты и оформлять сообщения в code-review
Сообщения к коммитам строятся по такой модели:
<br />
**[task] type(object): message**
<br />

* **task**. Номер задачи. Например, DataOps-2345, dto-2345. Если задачи нет, то можно опустить
* **object**. Что меняется.
    * если отчёт, то номер отчёта
    * если таблица, то имя таблицы
    * если плагин, то имя функции
    * если даг, то идентификатор дага (или ту его часть, которой достаточно, чтобы понять, о каком даге речь)
    * если ручка api, то её ендпоинт (или ту его часть, которой достаточно, чтобы понять, о какой ручке речь)
    * если коммит вносит изменения в несколько объектов одновременно, то либо их общий префикс, либо object можно опустить
* **type**. Каким образом меняется. Варианты:
    * `new` - создание
    * `rm` - удаление
    * `fix` - исправление ошибки 
    * `feat` - добавление/удаление колонки, признака, фильтра. Изменение, касающееся продуктовой ценности дага/отчёта/таблицы - того, что отдаётся в качестве результата
    * `ref` - рефакторинг кода, при этом конечный результат не меняется. Например, когда внедряется принципиально иной подход к решению задачи, оптимизируется код
    * `chore` - когда меняются коннекты, используемые утилиты, библиотеки, имена таблиц. Код не претерпевает принципиальных изменений, конечный результат остаётся таким же, как прежде
    * `style` - когда меняются css-стили (для отчётов), форматирование кода, комментарии к коду
    * если в коммите происходят изменения нескольких типов, то нужно выбрать наиболее значимое изменение
* **message**. Подробнее о том, что происходит в коммите - в нескольких словах. Можно опустить, если type - `new` или `rm` (в этом случае и так всё ясно)

Примеры:
* [dwh#232] new: ch4_to_dmo_waysheet_and_srid
* [DataOps-13081] feat(002_shk_for_lawyers): add currency field
* [DataOps-12370] fix(defect): empty last_shk_state
* [dto-3407] chore(srids_from_darkstore): positions v2 => v3
* [dto-3439] chore(001.1.4): from dm3 to ch13
* [DataOps-13242] rm: datamart.mp_available_stock_suppliers

Как оформляется сообщение в [code-review](https://band.wb.ru/dataops/channels/code-review):
- Сначала тегаем 
    - @dataops-dags для дагов и плагинов
    - @dataops-click для изменений в схеме таблиц
    - @kravcov.artemiy @inchenko.ilona для изменений в superset
    - Дополнительно - владельца дага или отчёта (когда не вы владелец, но вы вносите изменения в чужой продукт)
- Затем оставляем ссылку на задачу в формате markdown. Например, `[dto-3407](https://youtrack.wildberries.ru/issue/dto-3407/Dorabotka-api-po-problemnomu-tovaru)`
- Если прикладываете скрипт, который надо выполнить, то:
    - скрипты длиннее трёх строк или с секретами - через [privatebin](https://privatebin.wildberries.ru/)
    - скрипты в три строки или короче - отформатировать как multiline code (добавить три косые черты до и после кода)
    - кидать хоста перед ссылкой на скрипт (так же, как в db-grants)
- Если нужно только принять МР, но не выполнить - обязательно напишите об этом рядом с МР-ом (не в треде, не в личке). То же самое - если принять нужно в какое-то определённое время

# Использование вкладки Commits в приложении

## Структура интерфейса

### 1. Блок "Данные задачи"

| Поле | Описание |
|------|----------|
| Task Link | Ссылка на задачу (например, `https://youtrack.wb.ru/issue/DataOps-3326`) |
| Task ID | Автоматически извлекается из ссылки (readonly) |

### 2. Блок "Объект"

| Поле | Описание |
|------|----------|
| Type | Тип коммита: `fix`, `new`, `rm`, `feat`, `ref`, `chore`, `style` |
| Object category | Категория: `отчет`, `таблица`, `плагин`, `даг`, `ручка апи`, `несколько` |
| Object | Название объекта (с подсказкой в зависимости от категории) |

**Условные поля (зависят от категории):**

| Категория | Поля |
|-----------|------|
| отчет | Тестовый отчет, Продовый отчет, Перенести коннект |
| даг | Тестовый даг |

### 3. Блок "Формирование коммита"

| Поле | Описание |
|------|----------|
| Message | Текст сообщения коммита |
| Результат | Итоговая строка коммита (readonly) |
| Копировать коммит | Копирует строку в буфер обмена |

### 4. Блок "Сообщение в чат"

| Поле | Описание |
|------|----------|
| Теги | Выбор тегов из выпадающего списка |
| Выбранные теги | Отображает добавленные теги через пробел (readonly) |
| Очистить | Удаляет все выбранные теги |
| MR | Ссылка на Merge Request |
| Превью сообщения | Предпросмотр итогового сообщения |
| Копировать сообщение | Копирует сообщение в буфер обмена |

## Формат превью сообщения

```
#тег1 #тег2
[TASK-ID](ссылка_на_задачу)
MR: ссылка_на_mr
даг: [тест](ссылка)                           # для категории "даг"
отчеты: [тест](ссылка), [прод](ссылка)        # для категории "отчет"
надо перенести коннект: название_коннекта     # для категории "отчет"
```

## Управление тегами

Управление тегами находится в **Settings → Commits**:

1. Открыть Settings (кнопка в правом верхнем углу окна)
2. Перейти на вкладку "Commits"
3. Добавить новый тег: ввести название → нажать "Добавить"
4. Удалить тег: выбрать в списке → нажать "Удалить выбранный тег"

## Настройки окна

В **Settings → General** можно настроить:

- **Window width** - ширина окна приложения (400-1200 px, по умолчанию 600)

## Примеры использования

### Пример 1: Фикс отчета

1. Вставить ссылку на задачу в поле Task Link
2. Выбрать Type: `fix`, Category: `отчет`
3. Ввести номер отчета в поле Object (например, `004.1`)
4. Вставить ссылку на тестовый отчет (продовый заполнится автоматически)
5. При необходимости заполнить поле "Перенести коннект"
6. Добавить теги (@dataops-click, @kravcov.artemiy и т.д.)
7. Вставить ссылку на MR
8. Написать сообщение коммита
9. Скопировать коммит и сообщение в чат

### Пример 2: Новый даг

1. Вставить ссылку на задачу
2. Выбрать Type: `new`, Category: `даг`
3. Ввести идентификатор дага в поле Object
4. Вставить ссылку на тестовый даг
5. Добавить тег @dataops-dags
6. Скопировать результаты
