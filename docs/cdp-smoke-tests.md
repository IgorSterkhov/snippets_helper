# Smoke- и CDP-тесты desktop frontend

Этот документ объясняет, как устроены быстрые браузерные тесты для
desktop-frontend части приложения.

Главный файл:

```bash
desktop-rust/src/dev-test.py
```

Обычный запуск:

```bash
cd desktop-rust/src
python3 dev-test.py
```

Если все хорошо, в конце будет строка вида:

```text
=== 14/14 passed ===
```

## Что такое smoke-тест

Smoke-тест - это короткая проверка, что основные части приложения вообще
работают после изменения кода.

Название историческое: как будто мы включили устройство и проверили, не идет
ли из него дым. В нашем случае это значит:

- приложение открывается;
- основные вкладки загружаются;
- важные кнопки и модальные окна реагируют;
- простые пользовательские сценарии проходят без ошибки.

Smoke-тесты не заменяют полное ручное тестирование и не доказывают, что в
приложении нет ошибок. Их задача другая: быстро поймать грубые поломки,
например "вкладка больше не открывается" или "кнопка больше ничего не делает".

## Что такое CDP

CDP означает Chrome DevTools Protocol.

Проще говоря, это способ управлять Chrome программно. Обычно человек открывает
страницу, кликает мышкой и смотрит результат глазами. CDP позволяет скрипту
сделать почти то же самое:

- открыть страницу;
- нажать кнопку;
- заполнить поле;
- прочитать текст на экране;
- проверить, появился ли нужный элемент.

В наших тестах Chrome запускается без видимого окна. Такой режим называется
headless, то есть "без головы": браузер есть, страница загружается, JavaScript
работает, но окно на экране не показывается.

## Что именно запускает `dev-test.py`

Скрипт делает четыре вещи.

1. Запускает локальный web-сервер.

   Он отдает файлы из папки `desktop-rust/src` как обычный сайт.

   Адрес:

   ```text
   http://localhost:8765/dev.html
   ```

2. Запускает Chrome без окна.

   Chrome стартует с включенным удаленным управлением на порту `9222`.

3. Подключается к Chrome через CDP.

   После подключения тестовый скрипт может выполнять JavaScript внутри страницы.

4. Выполняет набор проверок.

   Каждая проверка печатает `PASS` или `FAIL`.

## Почему используется `dev.html`, а не настоящее Tauri-приложение

Desktop-приложение собрано на Tauri. В обычном режиме frontend вызывает native
команды Rust через `window.__TAURI__.core.invoke(...)`.

В браузере таких native команд нет. Поэтому для тестов используется файл:

```bash
desktop-rust/src/dev-mock.js
```

Он создает поддельный объект `window.__TAURI__` и отвечает на команды
frontend-а заранее подготовленными данными.

Например, если интерфейс вызывает команду `list_repos`, mock возвращает список
репозиториев из тестовых данных. Если интерфейс вызывает `add_repo_group`, mock
добавляет группу в браузерное хранилище.

Это дает быстрый способ проверить frontend без запуска Rust backend, базы
данных и настоящей синхронизации.

## Что такие тесты проверяют хорошо

CDP smoke-тесты хорошо подходят для проверки frontend-поведения:

- открывается ли вкладка;
- появляется ли модальное окно;
- работает ли кнопка Confirm;
- обновился ли список после действия;
- не сломалась ли загрузка JavaScript-модулей;
- есть ли нужный текст или элемент на странице.

Примеры текущих проверок:

- mock API установлен;
- статусная строка показывает frontend version;
- вкладка Exec показывает категории;
- модальное окно создания категории не закрывается при пустом имени;
- после заполнения имени категория появляется;
- template picker открывается и заполняет команду;
- вкладка Repo Search умеет создавать и удалять группы;
- fullscreen-карточка файла открывается и закрывается.

## Что такие тесты не проверяют

Важно понимать ограничения.

Эти тесты не проверяют:

- настоящую SQLite-базу приложения;
- реальные Rust/Tauri-команды;
- реальный sync с сервером;
- сборку `.dmg` или `.exe`;
- работу операционной системы, tray, hotkeys и native updater;
- внешний интернет, SSH, VPS и другие реальные интеграции.

Если ошибка находится в Rust backend или в серверной синхронизации, CDP
smoke-тест может ее не увидеть. Для таких случаев нужны Rust-тесты, API-тесты
или ручная проверка настоящего приложения.

## Как читать результат

Пример хорошего результата:

```text
PASS  T1 mock installed
PASS  T2 status bar shows v0.9.5-f1
PASS  T3 Exec tab renders categories

=== 14/14 passed ===
```

Это значит, что все 14 сценариев прошли.

Пример плохого результата:

```text
FAIL  T5 Confirm with name creates category: cats: ['System', 'Docker']

=== 13/14 passed ===
```

Это значит, что конкретный сценарий не увидел ожидаемый результат. В примере
тест ждал новую категорию, но в списке ее нет.

Если появляется `FATAL`, значит тестовая среда не смогла нормально стартовать.
Частые причины:

- Chrome не установлен или называется не `google-chrome`;
- порт `8765` или `9222` уже занят;
- в Python нет пакета `websockets`;
- страница упала на ранней JavaScript-ошибке.

## Как устроен один тест внутри файла

Упрощенно тест выглядит так:

```python
async def t3():
    await cdp.eval(
        "document.querySelector('.tab-btn[data-tab-id=\"exec\"]').click()"
    )
    await wait_until(
        cdp,
        "!!document.querySelector('#exec-cat-list .cat-name')",
        timeout=4,
    )
    cats = await cdp.eval(
        "[...document.querySelectorAll('#exec-cat-list .cat-name')].map(x => x.textContent)"
    )
    assert 'System' in cats, f'got cats: {cats!r}'
```

Что здесь происходит:

1. Скрипт нажимает кнопку вкладки Exec.
2. Ждет, пока на странице появится список категорий.
3. Читает названия категорий.
4. Проверяет, что среди них есть `System`.

Если проверка не проходит, Python выбрасывает ошибку, а общий runner печатает
`FAIL`.

## Зачем нужен `wait_until`

Интерфейс не всегда обновляется мгновенно. После клика JavaScript может
загрузить данные, перерисовать список или открыть модальное окно с небольшой
задержкой.

Функция `wait_until` решает эту проблему: она несколько раз проверяет условие,
пока оно не станет истинным или пока не выйдет время ожидания.

Без `wait_until` тесты были бы нестабильными: иногда проходили бы, иногда
падали бы просто потому, что проверка случилась слишком рано.

## Как добавить новый smoke-тест

Обычно порядок такой.

1. Добавить или проверить mock-данные в `desktop-rust/src/dev-mock.js`.

   Если frontend вызывает новую команду Tauri, для нее нужен handler в mock.
   Иначе тестовая страница напечатает предупреждение:

   ```text
   [mock] Unhandled command: command_name
   ```

2. Добавить новую функцию внутри `run_tests()` в `dev-test.py`.

   Лучше давать понятное имя и короткий сценарий: одно пользовательское
   действие или одна проверяемая проблема.

3. Вызвать ее через `await check(...)`.

   Пример:

   ```python
   await check('T15 pin task updates pinned strip', t15_pin_task_updates_strip)
   ```

4. Запустить:

   ```bash
   cd desktop-rust/src
   python3 dev-test.py
   ```

5. Убедиться, что новый тест сначала ловит проблему, а после исправления
   проходит.

## Пример для бага с Pin в Tasks

Для проблемы с закреплением задач хороший smoke-тест должен проверить именно
видимое поведение:

1. Открыть вкладку Tasks.
2. Убедиться, что в верхней панели есть chip закрепленной задачи.
3. Открыть эту задачу.
4. Нажать кнопку Pin, чтобы снять закрепление.
5. Проверить, что chip исчез из верхней панели.
6. Нажать Pin у другой задачи.
7. Проверить, что новый chip появился.

Такой тест полезен, потому что он проверяет не только сохранение поля
`is_pinned`, но и обновление верхней панели после действия пользователя.

## Когда запускать

Для frontend-изменений в `desktop-rust/src` желательно запускать:

```bash
node --check path/to/changed-file.js
cd desktop-rust/src
python3 dev-test.py
```

`node --check` быстро проверяет синтаксис JavaScript-файла.

`dev-test.py` проверяет, что приложение в браузерном mock-режиме все еще
проходит основные пользовательские сценарии.

Перед frontend-only OTA-релизом `dev-test.py` должен проходить полностью.

## Как это связано с настоящим приложением

Можно думать об этих тестах так:

- `dev-test.py` проверяет frontend быстро и дешево;
- Rust unit tests проверяют native backend;
- ручная проверка настоящего приложения нужна для мест, где важны Tauri,
  операционная система, sync, updater или реальные внешние сервисы.

Ни один слой не заменяет все остальные. Но CDP smoke-тесты хорошо закрывают
частую проблему: "мы поправили одну вкладку, а незаметно сломали открытие
модалки или обновление списка".
