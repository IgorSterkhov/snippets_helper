10. **UTF-8 safety при работе со строками пользовательского контента.** Любая обрезка/срез строк (заголовки заметок и задач, текст чекбоксов, SQL-тексты, имена тегов, template-тексты) — ВСЕГДА через char-индексацию (`val.chars().take(n).collect::<String>()`), **не** через байтовый slice (`&val[..n]`). Кириллица = 2 байта/символ, эмодзи 3–4 байта; срез посреди многобайтового символа вызывает Rust panic и отравляет Mutex. На каждую функцию, обрезающую строку, добавь регрессионный тест с кириллическим/эмодзи-инпутом. Этот баг уже положил sync в v1.3.0 (`sync/client.rs:389`) — не повторять.

11. **Poisoned mutex recovery в backend-командах.** В новых `#[tauri::command]`, работающих с БД, использовать `state.lock_recover()` (определён в `desktop-rust/src-tauri/src/db/mod.rs`), **не** старый паттерн `state.0.lock().map_err(|e| e.to_string())?`. SQLite-транзакции атомарны, poisoned-флаг на Rust-уровне безопасно сбрасывать. Старый паттерн означает: одна паника в любой команде → все последующие `lock()` валятся с `"poisoned lock: another task failed inside"`, включая auto-updater, и приложение превращается в кирпич до переустановки. v1.3.1 зачистил 107 сайтов — держим новый код консистентным.

12. **Native vs frontend-OTA version skew.** При добавлении новых `#[tauri::command]` (новые методы, расширение сигнатур, изменение типов параметров) — выкатывать только через `v*`-релиз (нативка + фронт вместе в одном теге). **Не** через `f-*` OTA. Если пушнуть фронт-OTA с вызовами новых команд раньше нативки, на старом `.exe` вызовы валятся с `"command not found"` (в плохом раскладе — ещё и panic → poisoned mutex, см. #11). `f-*` OTA — только для чисто-UI правок, которые не трогают IPC-surface (не добавляют/меняют invoke-имена).

13. **Rigorous review ключевых моментов на Opus 4.7 max-effort.** Перед коммитом/мержем: **merge-коммиты** через несколько версий вперёд (разрешение конфликтов в 3+ файлах), **массовые рефакторинги** (>50 сайтов, массовые замены паттернов), **спеки и implementation-планы** перед кодированием, **хотфиксы в critical path** (DbState, sync, auto-updater, release CI), **пре-релизные `v-*` билды** — прогоняй через subagent с `model: opus` и прямой инструкцией «think deeply / take your time / use extended reasoning». Передавай агенту точные пути к файлам + чек-лист того что верифицировать (API-сигнатуры, integration points, edge cases, CLAUDE.md §10-§12 compliance, contradictions между ветками/файлами). Агент возвращает blocking issues с `file:line`, почему это блокирует и предлагаемый фикс. `cargo check` + unit-тесты НЕ ловят конфликт-резолвы с потерянными строками, silent-antipattern creep в отдельной функции и race conditions в async-коде — Opus ловит. Применяй паттерн **автоматически** на каждом таком событии, без напоминания от человека.

14. **Дисковая гигиена для Rust-проектов и git-worktree.** Rust `target/` в `desktop-rust/src-tauri/` раздувается до 5–25 ГБ на каждый worktree (worktree'ы шарят `.git`, но **не** build-артефакты — у каждого свой `target/`). Диск на этом хосте 116 ГБ.
   - **Перед стартом новой крупной задачи** (создание worktree / чекаут в новый dev-directory) — глянь `df -h /` и `du -sh /home/aster/dev/*/target /home/aster/dev/*/desktop-rust/src-tauri/target 2>/dev/null | sort -hr | head`. Если свободного <15 ГБ — сначала чистим.
   - **При abandon / merge / долгом простое worktree** (>2 дней без активной работы) — `cd <worktree> && cargo clean`. Освобождает 4–8 ГБ моментально. Восстанавливается при следующей сборке, только время (~5 мин).
   - **Старые копии проекта** (типа `keyboard_helper_rust` — прежний rewrite) — держать в clean-состоянии, `target/` удалять сразу, если проект не активный.
   - **Не держать одновременно больше 3–4 worktree с горячими target/** — сложится 20+ ГБ только на artefact'ах.
   - **Никаких `sudo journalctl --vacuum`, `docker system prune`, `rm -rf ~/.gradle`** без явного подтверждения от пользователя — это разрушительные действия на shared-состоянии хоста.

15. **Авторелиз после завершения фичи.** Когда разработка пользовательской фичи или существенного поведенческого изменения завершена и все коммиты уже сделаны — **автоматически**, без отдельной просьбы, прогоняй полную релизную последовательность из `desktop-rust/RELEASES.md`:
    1. **Выбрать тип тега** по `RELEASES.md §1` (см. также CLAUDE.md §12):
       - **`v-*`** — изменения в `desktop-rust/src-tauri/` (нативка, БД-миграции, новые/изменённые `#[tauri::command]`'ы, bump major/minor/patch).
       - **`f-*`** — чисто-фронт-правки (`desktop-rust/src/`) **без** изменения IPC-сигнатур.
    2. Для `v-*` — bump версии в `desktop-rust/src-tauri/Cargo.toml` + `desktop-rust/src-tauri/tauri.conf.json` (одинаковое значение), refresh `Cargo.lock` через `cargo check`, добавить секцию сверху в `desktop-rust/CHANGELOG.md` (§9), коммит «(vX.Y.Z)»-стиля. Для `f-*` — версия не bump'ается, только коммит фронт-правок.

    4. `git push` ; `git tag <тег>` (для `v-*` — `vX.Y.Z`; для `f-*` — `f-YYYYMMDD-N`, где N инкрементируется при нескольких релизах за день) ; `git push origin <тег>`.
    5. Запустить `gh run watch <run_id>` в фоне, по завершении — проверить ассеты на тег-странице (5–7 для `v-*`, 3 для `f-*`) и smoke-curl `frontend-version.json` из `RELEASES.md §2.4`. Сообщить пользователю результат.

    **Когда правило НЕ применять автоматически** (явно спросить у пользователя — без явного «да» не пушить тег):
    - Чисто-технические изменения без user-facing поведения: рефакторинг, тесты, CI-скрипты, doc-only фиксы (там как правило релиз вообще не нужен).

    - Незакоммиченные изменения в неcвязанных файлах (риск утянуть в коммит чужое).
    - Срабатывание сразу после `merge`-коммита через несколько версий вперёд — там сначала ревью на opus (§13).
